# V2 participant host contract

The participant host is the single boundary between an already-made attention
decision and one normal participant act-or-silence turn. It never asks whether
the participant should respond again and never reclassifies an action at send
time.

## Turn lifecycle

1. Validate the complete attention decision, exact request correlation, policy
   consistency, and participant-budgeted snapshot before any branch—including
   SUPPRESS.
2. SUPPRESS stops without invoking the participant. WAKE, DEFER, trusted
   pre-attention bypass, and eligible operational error create one participant
   turn. An operator `NO_WAKE` error policy stops with a durable host receipt.
3. The turn declares available `message`, `reaction`, `privileged`, and
   `context-expansion` capabilities from actual host seams. Absence is explicit.
4. Context expansion accepts only the packet's opaque continuation handle. Each
   I-010D request and page is request/participant/room/scope/trigger/direction
   bound, byte/event bounded, coverage-checked, and subject to an eight-call
   absolute ceiling. The capability closes when the callback returns or fails.
5. Message reply/mention and reaction targets must name facts delivered in the
   initial packet or a validated expansion page.
6. The participant returns one concrete action or `None`. Unsupported meta
   answers are invalid. Privileged actions use the separate authorization seam.

## Receipt and action ordering

The host writes its immutable `participant-host` stage before transport can
write the later `transport` stage. For silence it attests `silent`. For a
selected ordinary action it truthfully attests `unknown`, because the native
effect has not yet occurred; the transport stage owns sent/failed/unknown
delivery truth. Unknown host-receipt persistence is an error and stops before
an ordinary action effect.

Attention-receipt persistence failure is infrastructure failure, not an
ordinary classifier error. The live runtime stops before participant invocation
and external effects. Observation ingestion and scheduler submission share one
outer lock so concurrent delivery cannot invert authoritative order or promote
an older event as the newest pending opportunity.
