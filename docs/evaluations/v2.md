# Evaluating the Nunchi V2 candidate

> Candidate documentation. These commands do not establish current-product or
> release status until their exact installed candidate is accepted and verified.

Nunchi separates deterministic mechanics from stochastic social behavior. A
unit test can prove that only a participant-shaped model may return social
`SUPPRESS`, that uncertainty widens to `DEFER`, and that a burst coalesces to the
newest pending opportunity. It cannot prove that a model consistently reads a
real room well. Conversely, an attractive transcript cannot prove identity,
authorization, receipt ownership, restart continuity, or package provenance.

The S01-S18 catalog binds both evidence classes without conflating them:

```sh
python3 -m evals.v2.parity.runner --list
python3 -m evals.v2.parity.runner --deterministic-time
python3 -m evals.v2.parity.runner --scene S17 --scene S18
python3 -m evals.v2.parity.runner --require-complete
```

Installed-surface and residue evidence is produced separately:

```sh
python3 -m evals.v2.provenance.runner --deterministic-time
python3 -m evals.v2.provenance.runner --install --output <new-evidence-path>
```

The second command builds and installs the exact wheel in a fresh environment;
it is not an editable/source-tree proxy. See
[`../product-surface-v2.md`](../product-surface-v2.md) for the deliberate
keep/migrate/remove table.

The runner reuses exact deterministic tests; it does not reimplement the
product or define a deterministic “correct social verdict.” Every record lists
the external evidence still required. A mechanics pass with `complete: false`
is intentionally incomplete, not a waiver. S12 is installed provenance and V1
residue removal; S14 is the six-stage live mixed-room ladder, so neither can be
passed by the offline runner.

## Binding external evidence to the frozen candidate

External evidence becomes machine-recognized only through the committed
`evidence/v2/parity/external-evidence-index.json`. Its closed root is:

```json
{
  "schema_version": 1,
  "candidate_commit": "<full frozen commit>",
  "records": [
    {
      "evidence_id": "<exact ID listed by the scene catalog>",
      "candidate_commit": "<same full frozen commit>",
      "status": "PASS",
      "recorded_at": "<UTC timestamp>",
      "attestations": [{"identity": "<reviewer/operator>", "family": "<model or human family>"}],
      "artifacts": [{"path": "evidence/v2/<record>", "sha256": "<digest>"}],
      "limitations": []
    }
  ]
}
```

The index and every artifact must be bounded regular files below `evidence/v2`,
must be tracked at the clean current commit, and must match their SHA-256
digests. The frozen candidate must be an ancestor of that commit, and every
intervening change must be evidence-only. A source, schema, test, evaluation,
packaging, or documentation change invalidates the index: reviewers must assess
the exact successor and a new evidence-only commit must bind their result.
OpenAI/Codex-only attestations cannot satisfy the cross-family review ID.

Without `--require-complete`, parity retains its earlier behavior: nonzero
means deterministic mechanics failed, while missing external evidence remains
explicit in JSON. With `--require-complete`, any missing evidence ID is also a
blocking exit. The index attests provenance and binding; it does not replace
human inspection of whether a transcript or review actually supports its PASS.

## Live social evaluation

Live provider and room runs must use pinned candidate, policy, model, prompt,
and surface identities. Repeated scenes measure flicker rather than hiding it.
Post-hoc graders evaluate whether agents miss direct opportunities, talk over
resolved moments, emit meta-admission answers, all speak, all stay mute, or
revive stale branches. The runtime must not use those graders to filter,
rewrite, or reclassify participant output.

The repository-owned repeated-provider runner is:

```sh
python3 -m evals.v2.social.runner --list
python3 -m evals.v2.social.runner \
  --policy /absolute/path/to/nunchi-policy.json \
  --trials 7 \
  --output evidence/v2/live/social-provider-<run-id>.json
```

The output path is exclusive-create and is never overwritten. Each case records
the exact candidate commit/dirty state, policy provenance, provider and model,
digest of the endpoint rather than its raw value, credential presence rather
than the credential, exact prompt digest, non-secret configuration digest,
classifier-projection digest, raw validated judgment, effective routed
decision, duration, ordered trial distributions, and flicker transitions. The
product request currently pins temperature zero; repeated trials still reveal
provider/model nondeterminism and drift instead of assuming identical output.

Four high-risk false-suppression cases have explicit repeated-run constraints
on both the classifier and effective disposition: an exact direct request, a
later direct correction, an open group opportunity, and a direct-looking moment
with honest coverage gaps. The remaining cases are intentionally measurement
only: exact peer address, later resolution, referential name use, ordinary
flow, display-name collision, and room-wide address do not have one universally
correct disposition. Their distributions and reasons require post-hoc review.
This keeps acceptance pressure on false negatives without smuggling a social
keyword oracle into runtime code.

The runner always reports `post_hoc_review_required: true` and
`product_completion_claimed: false`. A zero exit status says only that provider
calls were valid and the narrow machine-checkable safety constraints held. It
does not grade the participant's later action, close the mixed-room scene, or
authorize release.

The freshness scene deliberately holds a participant turn open while more room
events arrive, including a later resolution. The expected mechanic is one
active opportunity plus at most one replaceable newest pending anchor. The
first participant sees the current tail and may stay silent; intermediate
events remain context rather than FIFO jobs. Restart/backfill restores context
without replaying wake work.

## Evidence grades

- **Deterministic mechanics**: offline contract, normalization, scheduling,
  routing, receipt, and authorization checks.
- **Installed probe**: exact wheel, commit, executable, configuration,
  interface versions, process identity, restart, and one bounded native action.
- **Live provider**: repeated social scenes with raw model result, effective
  disposition, receipts, model provenance, and flicker counts.
- **Live room**: native transcripts plus off-surface receipts and action/silence
  grades for the fixed room ladder.
- **Independent review**: a frozen exact candidate reviewed across model
  families; blockers fixed and the exact successor reviewed again.

Only the combined evidence can satisfy the V2 goal. Secrets, raw credentials,
continuation authority, approval evidence, and capability grants must never be
written into evaluation projections or committed transcripts.
