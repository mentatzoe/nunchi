# Nunchi V2 Observation (I-020A ObservationProviderV2@1)

**Status**: slice `020-v2-observation`, `src/nunchi/observation.py`. This
document describes authorized slice implementation, not current V1 product
behavior. V1 remains the current product until the atomic V2 merge is
verified as `CUTOVER_VERIFIED` on `main` (see `README.md`).

`I-020A` is the shared, transport-neutral observation provider consumed by
every native-surface slice (`050`, `060`–`090`) after they accept this
handoff. It normalizes transport-attested native events into stable actors
and literal relations, assembles bounded factual `AttentionRequestV2`
snapshots (`I-010A`), exposes an optional host-owned `ContextContinuationV2`
fetch seam (`I-010D`), and emits one immutable observation-stage
`AttentionReceiptV2` record (`I-010E`) per request. It makes no attention
judgment, invokes no participant, and creates no social ledger — see
`docs/contracts/nunchi-v2.md` for the full `I-010A`–`I-010E` contract
family this slice consumes.

## Identity binding (FR-002)

`self.actor_id` is bound once, from a transport- or host-attested identity,
at construction. No name, alias, role, or message text ever establishes
authorship — an alias collision with another observed actor never flips
self binding.

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil",
    actor_id="discord:9001",          # the only fact that establishes self
    names=["Vigil", "Aether"],         # loose descriptors; never authorship
    role="developer",
    platform="discord",
    room_id="42",
    continuity_scope_id="discord:room:42#2026-07",
)
assert provider.self_block()["actor_id"] == "discord:9001"
```

## Ingesting native events (FR-003, FR-004, FR-010)

`ingest()` takes one *native event input*: an explicit `disposition` of
`candidate-event` (the native event plus the transport's own authorization/
routing provenance) or `unroutable` (the transport's proof that no
routable candidate event exists). Authorization is never derived from
payload content — a `candidate-event` without `authorized: True` raises
rather than being inferred. `unroutable` is an exact closed three-field shape
(`delivery_id`, `disposition`, `reason`); candidate-only or unknown fields reject
instead of silently discarding a supplied event. Constructor self/room/
visibility facts are validated before mutable state is initialized, and a
candidate whose parseable timestamp would decrease authoritative retained order
rejects before delivery, event, actor, or timestamp state mutates.
The last parseable timestamp is a constant-size provider/continuity-lifetime
watermark: eviction of its source event, including eviction by later undated
events, cannot erase ordering authority or admit an older parseable timestamp.
The complete caller-owned native input is deep-copied before any field is read
or validated; validation and retention use only that private copy, and copy
failure raises `ObservationInputError` before provider state changes.

The only three mechanical no-wake outcomes are `duplicate-retained` (exact
delivery ID seen before), `self-retained-no-wake` (the event is exact-self-
caused — retained, not dropped), and `unroutable`. Every other authorized
event is `observed`. Deterministic hygiene never reasons about mentions,
replies, or apparent resolution.

Exact self-causation (FR-004, resolved in D020-01): for `message` and
`reaction` events, self-causation is `author_id == self.actor_id`. A
`membership` event has no `author_id`; it is self-caused, and therefore
`self-retained-no-wake`, only when its `caused_by_actor_id` exactly matches
`self.actor_id`. A membership event where self appears only as the passive
`subject_actor_id` (self was joined/left by another actor, or the cause is
unknown) remains ordinary `observed` — being acted upon is not the
participant's own action.

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
outcome = provider.ingest({
    "delivery_id": "discord:msg:555",
    "disposition": "candidate-event",
    "authorized": True,
    "event": {
        "id": "e1", "type": "message", "author_id": "discord:1001",
        "text": "hey @Vigil can you take the deploy?",
        "mentioned_actor_ids": ["discord:9001"], "mentions_room": False,
    },
    "actors": {"discord:1001": {"display_name": "Zoe", "kind": "human"}},
})
assert outcome == "observed"
```

`reaction` and `membership` events carry their own literal relations rather
than the message shape above. A `reaction` event's `target_event_id` names
the event it reacts to, and `operation` (`add`/`remove`) is a literal fact,
never a resolved "the reaction still applies" judgment. A `membership`
event's `scope`, `subject_actor_id`, and `change` (`join`/`leave`) describe
who moved and where; the optional `caused_by_actor_id` names who caused it
only when the transport supplies it — otherwise it is omitted honestly
rather than guessed.

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
provider.ingest({
    "delivery_id": "d1", "disposition": "candidate-event", "authorized": True,
    "event": {"id": "e1", "type": "message", "author_id": "discord:1001",
              "text": "deploy is up", "mentioned_actor_ids": [], "mentions_room": False},
    "actors": {"discord:1001": {"display_name": "Zoe", "kind": "human"}},
})
outcome = provider.ingest({
    "delivery_id": "d2", "disposition": "candidate-event", "authorized": True,
    "event": {
        "id": "e2", "type": "reaction", "author_id": "discord:1001",
        "target_event_id": "e1", "reaction": "\U0001f44d", "operation": "add",
    },
    "actors": {},
})
assert outcome == "observed"
snapshot = provider.snapshot(trigger_event_id="e2", max_events=10, max_bytes=65536)
reaction_event = snapshot["events"][-1]
assert reaction_event["target_event_id"] == "e1"  # literal relation, not resolved semantics
assert "timestamp" not in reaction_event           # unsupplied fact stays an honest omission
```

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
outcome = provider.ingest({
    "delivery_id": "d1", "disposition": "candidate-event", "authorized": True,
    "event": {
        "id": "e1", "type": "membership",
        "scope": {"kind": "room", "id": "42"},
        "subject_actor_id": "discord:1001", "change": "join",
    },
    "actors": {},
})
assert outcome == "observed"
snapshot = provider.snapshot(trigger_event_id="e1", max_events=10, max_bytes=65536)
membership_event = snapshot["events"][0]
assert membership_event["subject_actor_id"] == "discord:1001"
assert membership_event["change"] == "join"
assert "caused_by_actor_id" not in membership_event  # unknown cause stays an honest omission
```

A native event that is authorized and routable but fails normalization
(missing required fields, a dangling actor reference, a duplicate event ID)
raises `nunchi.observation.ObservationInputError` — an operational error,
never a silent drop or a social suppression decision.

## Assembling a bounded snapshot (FR-006, FR-007)

`snapshot()` always includes the trigger, then the trigger's exact relation
closure (its `reply_to_event_id`/`thread_root_event_id`/`target_event_id`)
where it fits, then nearby events outward from the trigger — under hard
`max_events`/`max_bytes` and optional `max_age_seconds` caps — preserving
authoritative (ingestion) order throughout. If the trigger event alone exceeds
`max_bytes`, no truthful trigger-containing document can satisfy the hard cap,
so `snapshot()` raises `ObservationInputError` instead of returning an
oversized request. `coverage.truncated_by` names every limit that actually
excluded a candidate from an accepted snapshot; `has_more_before`/
`has_more_after` report honest boundary omission and `has_gaps` reports a
non-contiguous selection, a retention-bound eviction, or a literal relation
target that is unavailable or excluded. A retained relation target excluded by
an event, byte, or age budget also preserves that exact cause in
`truncated_by`; an unavailable target reports the gap without inventing a
budget cause.

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
for i in range(1, 4):
    provider.ingest({
        "delivery_id": f"d{i}", "disposition": "candidate-event", "authorized": True,
        "event": {
            "id": f"e{i}", "type": "message", "author_id": "discord:1001",
            "text": f"message {i}", "mentioned_actor_ids": [], "mentions_room": False,
        },
        "actors": {},
    })
request = provider.snapshot(trigger_event_id="e2", max_events=1, max_bytes=65536)
assert request["events"][0]["id"] == "e2"
assert "events" in request["coverage"]["truncated_by"]
```

## Optional continuation (FR-008, FR-009)

A `ContinuationProvider` wraps one `ObservationProvider` and issues bounded,
opaque, participant/room/continuity/trigger-bound capabilities. Issuance also
binds a private `originating_event_ids` set copied from the originating
request; it never appears on the closed capability wire document, and any
fetched page overlapping that set rejects before cursor state commits. A fetch
request carries no inline binding fields (per `I-010D`); the host's actual
call context must be the exact closed four-field `bound_to` object at fetch
time. Missing, malformed, wrong-valued, or additional properties reject before
any event is served or cursor state changes. Expiry, direction authorization,
caps, and cursor provenance are checked in the same fail-closed transition.
The complete fetch request and host context are deep-copied once at entry and
only those validated copies are used. Provider and continuation transitions
share one provider-owned re-entrant lock, so one-shot consumption, revocation,
and hard resource limits remain atomic under concurrent callers. All
`ContinuationProvider` wrappers over one provider also share one provider-owned
capability/cursor registry and must declare identical limits; wrapper
multiplicity cannot multiply capacity. A fixed 65,536-bit issued-handle filter
prevents generated handle reuse after either live issuance or revocation, so old
capabilities cannot resurrect. Collisions retry and fail closed after 16
attempts; the filter is constant-size and has no false negatives. A fresh
fetch at active-cursor capacity rejects before retained-event traversal or
continuation window construction. Known bounded-retention eviction is disclosed
as `coverage.has_gaps=true` on every continuation page.
`nunchi.observation.ContinuationError` is raised on any
binding/expiry/cap/cursor/merge-identity violation.

```python
from nunchi.observation import ContinuationProvider, ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
provider.ingest({
    "delivery_id": "d1", "disposition": "candidate-event", "authorized": True,
    "event": {"id": "e1", "type": "message", "author_id": "discord:1001",
              "text": "hi", "mentioned_actor_ids": [], "mentions_room": False},
    "actors": {},
})
continuation = ContinuationProvider(provider)
capability = continuation.issue(
    trigger_event_id="e1", originating_event_ids=["e1"],
    max_events_per_fetch=10, max_bytes_per_fetch=8192,
)
page = continuation.fetch(
    {"request_id": "req-1", "handle_id": capability["handle_id"],
     "direction": "before", "max_events": 5, "max_bytes": 8192},
    host_context=capability["bound_to"],
    fetch_time="2026-07-17T01:30:00Z",
)
assert page["handle_id"] == capability["handle_id"]
```

## The observation-stage receipt (FR-015)

`build_observation_receipt()` emits exactly one immutable `I-010E` record,
`stage="observation"`, `writer="observation-provider"`, correlated by
`request_id`. Its body carries only snapshot/coverage facts this provider
can attest — `schema_version`, `trigger_event_id`, `continuity_scope_id`,
`event_count`, `byte_count`, `coverage`, `included_event_ids` — and nothing
from a later stage. It never adds an estimated-token field: the accepted
`I-010E` observation body is closed with no token field (see "Token-size
proxy" below).

The provider privately retains a deep copy of each issued snapshot awaiting
attestation. Receipt construction copies the complete caller request before
validation, compares that stable copy to private issued authority, and computes
every attested field from the private issued document. It accepts that exact
document once; unknown, mutated, fabricated, copy-failed, and already-attested
requests reject. Concurrent or failed mutation
attempts do not consume the rightful pending request. Pending and recently
attested request-ID registries are each bounded by `max_pending_receipts`
(default 1024). At the pending cap, creation of a new snapshot rejects before
return, so an already-issued request never loses its promised receipt. A fixed
65,536-bit SHA-256 membership filter permanently prevents request-ID reuse per
provider with bounded memory and no false negatives; collision/saturation can
only reject a fresh ID, which fails closed. The recent-attestation registry is
bounded diagnostic state, not the uniqueness authority.

This slice consumes the accepted `I-010E AttentionReceiptV2@2`
amendment (`evidence/v2/observation/dependency-010-amendment-A1-acceptance.md`);
the amendment changes only the separately owned `attentionBody`, and the
observation body mirrored above is proven byte-for-byte unchanged from `@1`
— no implementation change was owed by the version rebind.

```python
from nunchi.observation import ObservationProvider

provider = ObservationProvider(
    participant_id="vigil", actor_id="discord:9001",
    platform="discord", room_id="42", continuity_scope_id="discord:room:42#2026-07",
)
provider.ingest({
    "delivery_id": "d1", "disposition": "candidate-event", "authorized": True,
    "event": {"id": "e1", "type": "message", "author_id": "discord:1001",
              "text": "hi", "mentioned_actor_ids": [], "mentions_room": False},
    "actors": {},
})
request = provider.snapshot(trigger_event_id="e1", max_events=10, max_bytes=65536)
receipt = provider.build_observation_receipt(request)
assert receipt["stage"] == "observation"
assert "estimated_tokens" not in receipt["body"]
```

## Token-size proxy (FR-013) — evidence only

`estimate_tokens(serialized_byte_size(document))` returns the slice-owned
`utf8-bytes-ceil-div4@1` proxy: `estimated_tokens = (serialized_utf8_bytes +
3) // 4`, always paired with `estimator_id`, `serialized_bytes`, and
`model_id: null`. It makes no model-tokenizer claim and is never written
onto `I-010E` (which is closed with no token field) — it lives only in
separate evidence (`evidence/v2/observation/*.jsonl`).

```python
from nunchi.observation import estimate_tokens, serialized_byte_size

document = {"hello": "world"}
proxy = estimate_tokens(serialized_byte_size(document))
assert proxy["estimator_id"] == "utf8-bytes-ceil-div4@1"
assert proxy["model_id"] is None
```

## Reference variants and the comparator (FR-011, FR-012) — reference only

`evals/v2/observation/capabilities/reference_provider.py` supplies
`restart-safe`, `session-only`, `unknown`, and `known-gap` reference
variants that simulate a restart and an honest backfill fraction — never a
real persistence layer. Each variant reports its continuity fact on
`coverage.continuity`, so a downstream consumer can read it without
inspecting the reference source:

- **`restart-safe`** — every persisted event is recovered after a
  simulated restart (full backfill). A consumer may treat history before
  the trigger as ordinarily available across a restart.
- **`session-only`** — a simulated restart recovers nothing; this is an
  honest session gap, never upgraded to `restart-safe` by inference. A
  consumer must not assume any pre-restart history survived.
- **`unknown`** — whether restart safety holds at all cannot be attested
  (the host itself cannot say either way); recovery behavior mirrors
  `session-only`, but the coverage fact stays explicitly `unknown` rather
  than being guessed as either safe or unsafe.
- **`known-gap`** — a partial backfill: some prefix of history is
  recovered, and the dropped tail is named in the reference's
  `known_gap_event_ids`. A consumer reading a `known-gap` result should
  treat the recovered prefix as available and the gap IDs as an honest,
  explicit omission — not silently absent and not worth re-requesting, since
  the reference has already declared they will never come back.

`evals/v2/observation/compare.py` compares two
`AttentionRequestV2` documents assembled from equivalent native facts and
classifies every difference as *explained* (an honestly declared capability
gap) or *unexplained* (a real divergence); zero unexplained differences is
the reusable comparison contract. It compares schema/self/room/actors,
authoritative event order and complete event fields, trigger or anchor,
semantic coverage and budget fields, continuation capability shape, direction,
and whether a continuation page has a successor. Request-local `request_id`,
issued `handle_id`, exact cursor bytes, and expiry-clock values are deliberately
opaque; their semantic presence and all non-opaque capability fields still
compare. Capability explanations must name exact unavailable event/actor IDs or
semantic paths; one-sided native facts are never silently accepted.

**These reference variants and the comparator prove reusable mechanics
only.** They establish no real-surface conformance, no restart-safety
claim, no social-suppression eligibility, and no cross-surface parity for
an installed transport. Each of slices `050` and `060`–`090` must run its
own real binding against the restart/backfill scene and the comparator
before claiming social-suppression eligibility for that surface; slice
`110` alone owns the final cross-surface parity claim.

## What this slice does not do

- No attention judgment, participant invocation, reply composition, or
  send-time safety (owned by slices `030`/`040`/`050`/`060`–`090`).
- No classifier-safe projection or redaction of the `I-010A` `continuation`
  capability — that is slice `030`'s sole responsibility in
  `src/nunchi/core.py`. This slice hands the expansion-availability input
  to `v2-core-owner` unchanged.
- No participant roster, handled/open ledger, obligation queue, or
  outcome-derived retention (FR-010); retention is bounded and outcome-
  neutral regardless of any later attention result.
- No edits to `010`-owned schemas under `schemas/v2/`.
