# Operating the Nunchi V2 candidate

> Candidate instructions. Do not present this branch as the verified current
> release before the recorded atomic cutover.

## Prepare trusted state

Create an owner-only receipt directory and copy the example policy to an
absolute owner-only file:

```sh
install -d -m 700 /absolute/path/to/nunchi-receipts
install -d -m 700 /absolute/path/to/nunchi-state
install -d -m 700 /absolute/path/to/nunchi-state/codex-home
install -d -m 700 /absolute/path/to/nunchi-state/codex-workspace
cp examples/v2/operator-policy.example.json /absolute/path/to/nunchi-policy.json
chmod 600 /absolute/path/to/nunchi-policy.json
```

Authenticate the dedicated Codex home once. Do not reuse the interactive
operator's main Codex home; this keeps room transcripts, authentication,
configuration, plugins, and continuity in an independently permissioned
boundary:

```sh
CODEX_HOME=/absolute/path/to/nunchi-state/codex-home \
HOME=/absolute/path/to/nunchi-state/codex-home \
codex login
```

Edit the participant ID, continuity scope, provider endpoint/model/key, receipt
directory, and grants. The policy parser is closed: unrecognized fields or
comments are errors. An empty grants array is the safe starting point.

`attention.preattention_enabled=false` is trusted bypass: it makes zero
classifier calls and wakes one advice-free participant turn. It is not a room
command. `social_suppression_enabled=false` only widens model `SUPPRESS` to
`DEFER`. `error_action` is `WAKE` by default or explicit `NO_WAKE`; neither is a
social disposition.

## Validate the portable attention path

Supply a valid `AttentionRequestV2` on stdin:

```sh
nunchi attention-v2 --config /absolute/path/to/nunchi-policy.json < request.json
```

Exit status is `0` for a valid decision or trusted bypass, `2` for invalid
input JSON, `3` for validation/configuration/budget errors, and `1` for
provider/runtime/receipt-sink failures.

## Start the shared Discord source

The V2 transport requires an exact room allowlist:

```sh
export NUNCHI_DISCORD_TOKEN='...'
export NUNCHI_MCP_DISCORD_MODE=v2
export NUNCHI_MCP_DISCORD_CHANNELS='152'
nunchi-mcp-discord
```

Optional `NUNCHI_MCP_DISCORD_BLOCKED_ACTORS` is a comma-separated list of exact
Discord user snowflakes. V2 notifications include humans, peer bots, and exact
self-authored events. Self events are retained for continuity and do not create
wake work. Message and reaction events are supported; membership visibility is
reported unavailable by this transport candidate.

## Start Codex presence

```sh
nunchi-codex-room-v2 \
  --policy /absolute/path/to/nunchi-policy.json \
  --channel-id 152 \
  --self-user-id 149 \
  --participant-id vigil \
  --participant-name Vigil \
  --session-path /absolute/path/to/nunchi-state/codex-room-session.json \
  --codex-home /absolute/path/to/nunchi-state/codex-home \
  --participant-workspace /absolute/path/to/nunchi-state/codex-workspace
```

On startup, at most 100 recent Discord messages are restored as factual
context. They are not replayed as wake jobs. Live events use one active turn
plus one replaceable newest pending anchor; a burst cannot become a FIFO reply
queue. The Codex thread ID is persisted owner-only and resumed exactly.
The session path must be absolute; a symlink, permissive file or directory,
oversized file, or corrupt document fails closed before Codex starts.
The Codex home and participant workspace must already be absolute, owner-only
directories. The workspace should contain no operator files or credentials.

## Wire privileged approvals

Privileged actions are disabled unless the embedding host constructs one
`PrivilegedActionCoordinator` with an exact capability-to-executor map and a
durable `ReloadingPolicyAuthorizationSink`. The shared `LiveRoomRuntime`
exposes two host-only methods:

- `pending_privileged_actions()` returns an inspectable copy of each exact,
  unexpired proposal for an operator UI;
- `complete_authenticated_approval(attestation)` completes a proposal only
  after that UI or transport has authenticated the exact approver actor.

Do not connect either method to ordinary Discord messages, reactions, model
output, or quoted text. A platform adapter must authenticate the operator
outside the room-message path and construct the closed attestation with
`challenge_id`, unique `attestation_id`, exact `approver_actor_id`, trusted UTC
`approved_at`, and channel `authenticated-transport` or `local-operator`.
The coordinator then reloads policy and executes the retained digest-bound
operation once. Pending approvals expire in memory and disappear on restart;
operators must ask the requester to propose again instead of replaying them.

## Required verification

```sh
python3 -m unittest
python3 -m evals.v2.parity.runner --deterministic-time
python3 -m evals.verdict_suite.runner --list
python3 scripts/check_governance.py --check-cli
```

The parity runner is expected to report `candidate_complete: false` until its
listed installed, live-room, live-provider, and independent-review evidence is
attached. A zero exit status means the deterministic mechanics passed; it does
not convert those external obligations into passes. See
[`../evaluations/v2.md`](../evaluations/v2.md).

Before a real-room candidate is accepted, also demonstrate `SUPPRESS`, `WAKE`,
classifier `DEFER`, transition-margin `DEFER`, trusted bypass, provider error
recovery, restart continuity, replies, explicit mentions, reactions, silence,
no-FIFO burst coalescing, unauthorized action denial, approval, revocation and
replay rejection against the exact installed wheel/commit.
