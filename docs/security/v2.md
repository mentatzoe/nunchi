# Nunchi V2 security model

> Candidate documentation. V1 remains the verified current product until the
> atomic V2 cutover is accepted, merged, and verified on `main`.

Nunchi separates two questions that must never be delegated to the same model:

1. **Would this participant naturally contribute now?** A participant-shaped
   stochastic judgment returns `SUPPRESS`, `WAKE`, or `DEFER`.
2. **May this exact user cause this exact privileged effect?** Deterministic
   host code derives the requester from transport-attested event identity and
   checks a trusted capability policy immediately before execution.

Room prose, display names, aliases, quotes, mentions, model claims, and earlier
decisions answer neither identity nor authorization questions.

## Identity and routing boundary

Each transport binds one exact participant actor and an exact room allowlist
from trusted host configuration. Discord uses user/channel/message snowflakes,
Matrix uses user/room/event IDs, and Telegram uses numeric user/chat/message
IDs. Display names remain descriptive actor facts only.

An authorized, routable native event is retained before attention. The only
deterministic no-wake classes are an exact duplicate delivery, an exact
self-caused event, an authorization/routing rejection, or a delivery from which
no native event can be constructed. A malformed event after construction is an
operational error, never social `SUPPRESS`.

## Privileged action authorization

A privileged proposal must bind all of the following:

- one unique action ID;
- the exact originating event ID;
- a canonical SHA-256 digest of the exact operation;
- one namespaced capability, such as `workspace.file.write`;
- platform, room, participant, resource kind and resource ID;
- one declared impact class.

The guard then:

1. validates the proposal and the provider-issued observation;
2. derives the requester from the exact origin event's attested actor ID;
3. reloads the owner-only operator policy;
4. matches an active, unexpired grant with exact capability, scope, resource,
   impact and requester;
5. returns `ALLOW`, `DENY`, or `APPROVAL_REQUIRED`;
6. immediately before execution, reloads policy and rechecks the grant,
   approval, action binding, digest and operation;
7. consumes `ALLOW` once before dispatch, so replay cannot repeat the effect.

Replay bindings are retained for the life of the guard and never evicted to
make room for a new action. They are capacity-bounded; once full, new unique
privileged actions fail closed with `deny-unsupported-seam` until the host is
deliberately restarted. The in-process audit view is also a bounded diagnostic
window. Durable authorization evidence belongs in the owner-only receipt/audit
sink, not an ever-growing room-process list.

Approval challenges and evidence are host-only. A normal room message cannot
approve an action. Approval evidence must arrive through an authenticated
transport or a local operator channel, match the immutable challenge and an
explicit allowed approver, and remain unexpired and unused.

The shared host coordinator keeps the exact approval-bound request,
observation, operation and executor only in bounded, expiring memory. Its
operator view contains the exact operation and full host decision so an
approver need not approve blindly, but that view must only be exposed on the
authenticated operator surface. Pending proposals are intentionally lost on
restart and never reconstructed from room history. When the pending limit is
full, new privileged proposals fail closed; the host does not duplicate policy
matching to guess which might have been direct grants.

Every full authorization decision is written to the owner-only audit sink
before an effect. A direct effect also waits for the participant-host receipt.
If either persistence result is unknown, the one-use allow is abandoned and
the operation has zero effects. Approval completion writes a new full decision
before dispatch. Invalid approval attempts remain auditable but do not consume
the rightful pending challenge; expiration, successful completion and restart
remove it.

No grant means deny. High-impact actions are therefore unavailable by default;
an operator must deliberately configure direct execution or authenticated
approval for an exact requester and resource.

## Participant and transport confinement

The attention classifier receives no credentials, continuation handles,
cursors, expiry secrets, capability grants, or approval evidence. A woken
participant receives current bounded room facts and may return an actual room
action or silence; it does not receive a tool permission simply because it was
woken.

The Codex reference integration runs in a dedicated owner-only Codex home and
empty participant workspace. It ignores user configuration and project rules,
uses strict configuration, disables shell, unified execution, code mode,
browser, app, MCP-like, computer-use, multi-agent, plugin, hook, skill, and web
tool paths, disables login-shell and environment inheritance, and receives a
minimal subprocess environment that excludes Discord, classifier, and operator
service secrets. Read-only sandbox policy and no approvals remain defense in
depth, not the primary tool-isolation claim. The prompt is one JSON envelope;
untrusted room text cannot close a pseudo-XML delimiter. A strict final-output
schema permits only a message, reaction, or `null`, and the host performs
delivery. The host also parses the Codex JSONL event stream and rejects the
turn before persisting continuity or accepting an action if any tool, patch,
hook, search, permission, or subagent event appears. Discord closes
`allowed_mentions` and enables only the exact user IDs
named by the structured action, so arbitrary `@everyone` or role text remains
inert.

## Policy and receipt files

The operator policy must be an absolute-path regular file owned by the current
effective user, opened without following symlinks, with no group/other
permissions. Duplicate JSON keys, non-finite numbers, unknown fields, partial
objects, unsafe paths and oversized files fail closed. Its provenance is the
source label plus SHA-256 of the exact bytes; secrets and paths are not
projected to models.

The receipt directory must already exist, be owned by the effective user, and
have no group/other permissions. Each request/stage record uses exclusive
creation and is never overwritten. The stage order and sole writers are:

```text
observation-provider -> attention-engine -> participant-host -> transport
```

A participant-host action is persisted before transport delivery. The later
transport stage records `sent`, `failed`, `unknown`, or `unavailable` without
rewriting the host's fact.

Persistent participant identity is trusted state too. The Codex room-session
path must be absolute; its containing directory is owner-only, and the regular
state file is opened without following symlinks, owned by the effective user,
mode `0600`, size-bounded, shape-checked, and replaced atomically with a
directory sync. Unsafe or corrupt state stops that participant turn before a
Codex process is started. A room message can neither select nor reset the
persisted thread.

## Operator checklist

- Use a dedicated platform account and least-privilege token.
- Allowlist exact rooms; never use display names as routing policy.
- Start with an empty `authorization.grants` array.
- Prefer approval mode for destructive, secret-bearing, account-configuration,
  and external-side-effect capabilities.
- Keep policy and receipt paths owner-only (`0600` file, `0700` directory).
- Treat `unknown` receipt persistence or delivery as requiring investigation,
  not an automatic retry.
- Put `pending_for_operator()` and `complete_authenticated_approval()` behind a
  genuinely authenticated host surface. Never call the latter from ordinary
  room-message or reaction handlers.
- Never replay pending wake work after restart; restored history is context.
- Run the deterministic security suite and a bounded installed-runtime probe
  before enabling a real room.
